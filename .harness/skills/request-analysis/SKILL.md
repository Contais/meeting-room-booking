# 需求分析技能

## 概述
将模糊的业务需求转化为清晰的技术任务，识别业务影响范围和技术风险，输出标准化的需求分析报告。

## 触发条件
当收到新功能需求或变更请求（如 PRD 评审、用户故事创建）时，自动激活此技能。

## 执行步骤

### Step 1: 需求解析
- **提取核心功能点**：识别需求中的主语、谓语、宾语，确定核心业务动作。
- **识别功能边界**：明确需求涉及的前端页面、后端接口、第三方服务。
- **明确验收标准**：提取需求中定义的“完成定义”（DoD），如响应时间、数据准确性。
- **输出**：`需求清单.md`

### Step 2: 业务影响分析
- **识别受影响的业务模块**：映射到具体的领域模型（如用户、商品、订单）。
- **评估业务流程变更**：绘制变更前后的流程图，识别断点。
- **确定用户影响范围**：明确哪些角色（买家、卖家、管理员）会使用该功能。
- **输出**：`影响模块列表.md`

### Step 3: 技术影响分析
- **评估数据库变更需求**：判断是否需要新建表、新增字段或修改索引。
- **评估 API 变更需求**：检查现有 Controller/Service 是否需要修改或新增。
- **评估缓存/MQ 变更需求**：判断是否需要更新 Redis 缓存策略或发送新的消息队列事件。
- **评估前端变更需求**：判断页面组件、状态管理（Store）是否需要调整。
- **输出**：`技术变更清单.md`

### Step 4: 冲突检测
- **检查与现有功能的冲突**：例如新增的优惠券逻辑是否与现有的满减逻辑互斥。
- **检查与编码红线的冲突**：对照《编码规范.md》，检查需求是否存在违规设计（如直接操作数据库）。
- **检查与架构约束的冲突**：确认是否违反了微服务边界（如跨服务直接调用）。
- **输出**：`冲突报告.md`

### Step 5: 输出
生成需求分析报告，写入 `.harness/changes/{feat-name}/summary.md`，包含：
- 需求描述与验收标准
- 技术变更清单
- 业务影响范围
- 冲突与风险提示
- 任务拆分建议
