# 编码规范 (Coding Standards)

## 1. 通用原则

1. **清晰优于简洁**：代码必须易于理解，避免过度优化。
2. **单一职责**：每个类/方法只做一件事。
3. **防御性编程**：对外部输入进行严格校验。
4. **避免重复**：复用已有代码，遵循DRY原则。

## 2. 后端规范 (Java/Spring Boot)

### 2.1 命名规范
- **类名**：使用大驼峰命名法 (PascalCase)，如 `OrderServiceImpl`。
- **方法名**：使用小驼峰命名法 (camelCase)，如 `calculateTotalAmount`。
- **变量名**：清晰表达意图，避免缩写。
- **常量**：全大写，下划线分隔，如 `MAX_RETRY_COUNT`。

### 2.2 代码结构
- **Controller**：仅负责参数校验和响应封装，不包含业务逻辑。
- **Service**：核心业务逻辑，事务控制必须在此层。
- **Mapper**：数据访问，禁止在Mapper中写复杂业务逻辑。
- **Entity**：与数据库表一一对应，不包含业务逻辑。

### 2.3 代码风格
- **缩进**：4个空格。
- **行宽**：不超过120字符。
- **空行**：方法之间空一行，逻辑块之间空一行。
- **注释**：公共API必须添加Javadoc，复杂逻辑添加行内注释。

### 2.4 异常处理
- 使用自定义异常体系：`BusinessException`。
- 禁止在Service层捕获异常后不处理或直接抛出。
- 全局异常处理器必须捕获所有未被处理的异常。

### 2.5 事务管理
- 使用 `@Transactional` 注解管理事务。
- 必须指定 `rollbackFor` 属性，如 `@Transactional(rollbackFor = Exception.class)`。
- 避免在事务中调用外部HTTP接口。

### 2.6 测试规范
- 测试类命名：`{ClassName}Test`。
- 测试方法命名：`should_{expectedBehavior}_when_{condition}`。
- 必须包含正向测试和异常测试。
- 测试数据应使用 `@Sql` 或 `@BeforeEach` 初始化。

## 3. 前端规范 (TypeScript/Vue 3)

### 3.1 命名规范
- **组件名**：使用大驼峰命名法 (PascalCase)，如 `ProductCard.vue`。
- **文件名**：与组件名一致，如 `ProductCard.vue`。
- **变量/函数**：使用小驼峰命名法 (camelCase)。
- **常量**：全大写，下划线分隔。

### 3.2 Vue 3 规范
- 优先使用 `<script setup>` 语法。
- Props 必须定义类型，使用 `defineProps`。
- Emits 必须定义类型，使用 `defineEmits`。
- 组件样式使用 `scoped` 属性。

### 3.3 TypeScript 规范
- 避免使用 `any` 类型，必须明确定义类型。
- 使用接口定义对象结构，不使用类型别名 (`type`)。
- 函数必须声明返回值类型。

### 3.4 API 调用
- 统一使用 Axios 实例。
- 请求拦截器必须添加认证Token。
- 响应拦截器必须统一处理错误响应。
