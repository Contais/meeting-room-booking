# 开发流程规范 (Development Workflow)

## 1. 需求分析阶段

1. 接收需求后，首先阅读 `.harness/wiki/` 中的相关知识。
2. 识别受影响的功能模块和业务规则。
3. 输出 `.harness/changes/{feature-name}/summary.md`，包含：
   - 需求描述与验收标准
   - 技术变更清单
   - 冲突与风险

## 2. 技术方案设计

1. 根据需求复杂度，决定是否需要进行技术方案评审。
2. 核心功能必须输出技术方案文档。
3. 方案通过后，才能进行编码实现。

## 3. 编码实现阶段

1. 遵循 `.harness/rules/编码规范.md` 中的所有规定。
2. 按照 `.harness/skills/coding-skill/SKILL.md` 中的步骤进行编码。
3. 每完成一个子任务，及时提交代码，并写明提交信息。

### 3.1 分支管理
- 从 `develop` 分支创建特性分支：`feature/{feature-name}`。
- 开发完成后，提交 Pull Request 到 `develop` 分支。
- 禁止直接向 `main` 或 `develop` 分支推送代码。

### 3.2 提交规范
- 使用 Conventional Commits 规范：
  - `feat`: 新功能
  - `fix`: 修复bug
  - `docs`: 文档更新
  - `style`: 代码格式调整
  - `refactor`: 重构
  - `test`: 测试相关
  - `chore`: 构建过程或辅助工具的变动

## 4. 代码评审阶段

1. 提交PR后，自动触发CI流水线。
2. 至少一名团队成员进行代码评审。
3. 评审通过后，才能合并到 `develop` 分支。
4. 核心代码必须经过专家评审 (`.harness/skills/expert-reviewer/SKILL.md`)。

## 5. 测试阶段

1. 单元测试：开发者负责编写，CI强制执行覆盖率门槛。
2. 集成测试：测试团队负责，验证端到端流程。
3. 性能测试：高并发场景必须进行压力测试。

## 6. 部署上线

1. 从 `develop` 分支合并到 `main` 分支。
2. 自动触发生产环境部署流水线。
3. 部署后进行 smoke test，确保核心功能可用。
